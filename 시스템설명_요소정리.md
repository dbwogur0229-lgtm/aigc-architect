# AIGC Architect — 시스템 설명 요소 정리
### 선배 미팅 · 면접 · Notion 공용. 숫자는 2026-07-16 코드 기준 실측.

---

## 0. ⚠️ 용어부터 — "LLM을 만들었다"고 하지 말 것
- ❌ "제가 만든 LLM" → 파운데이션 모델을 만든 게 아님. 면접에서 감점 포인트.
- ⭕ "**LLM 기반 RAG 에이전트**(AI Product)를 만들었다" / "LLM을 **부품으로 조달**하고, 지식기반·검색·오케스트레이션·실행통제(Harness)를 직접 설계했다"
- 내가 만든 것: 룰북 + 코퍼스 + 청킹·검색(RAG) + 스킬·에이전트 + Harness + UI. 조달한 것: Claude API(추론), Voyage API(임베딩) — 둘 다 없어도 동작(폴백).

## 1. 한 줄 아키텍처
**입력**(ITGC 게이트 7체크 · 산업 · 규모 · 조달 · 프로세스) → **에이전트**(5스텝 오케스트레이션, 조건 분기) → **스킬 6개** 순차 실행 → **RAG 검색**(130청크 코사인 유사도) → **LLM 재기술**(있을 때만) → **Harness 3중 게이트** → 통과分만 UI 출력, 나머지는 격리·로그.

## 2. 지식 기반 (직접 구축, 이 프로젝트의 본체)
| 구성 | 내용 |
|---|---|
| 룰북 (YAML 2개) | `industry_profiles`: 5산업 × 13프로세스 유의성 매핑. `control_ledger`: ITGC 게이트 7체크(4카테고리) + deep 2프로세스(aipc 8개·ecl 6개) + outline 8프로세스 통제 |
| 근거 검증 | 통제별 evidence(기준서명·문단·verified 플래그). **18건 전량 원문 대조 완료** (신규 5건은 2026-07-16 검증 — 과정에서 출처 4건 교정: 1100→315 보론6 위임, 전수 근거 330→500 A64 등. 반대 문구도 룰북에 병기) |
| 코퍼스 (md 9개) | 감사기준서·K-IFRS **원문 대조 발췌** — 330·500·530·1100 / 540·705 / 315 보론6-2(ITGC)·보론3-20(통제활동 유형)·확장가능성 / 402(서비스조직) / K-IFRS 1109·1008 / 내부회계관리제도 개념체계 |

핵심 설계 사상: **기준서 지식을 LLM 파라미터가 아니라 사람이 검증한 파일에 둔다.** 감사 도메인은 "그럴듯함"이 아니라 "몇 조 몇 항"의 검증가능성이 생명이라서.

## 3. 청킹 — 총 130청크
- 룰북 유래 46개: 통제 27 + 리스크 16 + 핵심명제 2 + 논리붕괴 1 (YAML 항목 단위)
- 코퍼스 유래 84개: md 문단 단위 분할
- 구조: `Chunk{id, text, meta(process_id, kind, standard)}` — id로 실행로그에서 인용 추적 가능

## 4. 검색 (RAG)
- 임베딩: **Voyage API → 키 없으면 로컬 TF-IDF 폴백** (배포본은 폴백 모드로 돌아감)
- 검색: numpy 코사인 유사도. **벡터DB 미사용** — 130청크 규모에 벡터DB는 과잉, 의존성 최소화가 배포 안정성에 유리 (묻히면 "규모가 1만 청크를 넘으면 그때 도입" 답변)
- `grounded_sources_for(pids)`: 이번 실행에서 검색된 청크의 출처 집합 = **인용 게이트의 접지(grounding) 집합**

## 5. LLM의 역할 — 생성이 아니라 재기술
- Claude API, **temperature=0 고정** (비결정성 최소화 — 이 도구가 남에게 요구하는 통제를 스스로 지키는 첫 항목)
- 역할: 룰북 통제 원문을 조합 맥락(산업·규모·조달)에 맞게 **KSA 용어로 재기술**하는 것뿐. 통제 자체·근거 조항은 룰북에서만 나옴
- **API 키 없으면 룰북 design 원문 그대로 출력 = 지금 배포 상태.** "LLM이 빠져도 결론이 성립하고, LLM은 표현만 개선한다" → 할루시네이션이 결론을 오염시킬 경로를 구조적으로 차단

## 6. 에이전트 (agent.py)
- 제너레이터 기반 **5스텝** 진행 이벤트(UI에 실시간 표시): 게이트 확인 → 유의계정 식별 → 경영진주장-리스크 매핑 → 통제 검색·맥락화 → 감사인 관점 생성
- **조건 분기**: ITGC 실패 → 즉시 중단(층1 없이 층2·3 무의미) / `ai_performed_control` 선택 → aipc_01·02 강제 포함 + 비결정성 추가 검색 / 산업별 프로세스 상이

## 7. Harness — 자기참조 설계 (최대 차별점)
"이 도구가 기업에 설계해주는 AIGC 통제를, 도구의 실행기가 스스로에게 적용한다."
| 게이트 | 내용 | 대응 AIGC 통제 |
|---|---|---|
| ① 스키마 검증 | pydantic — 필수 필드 누락 시 폐기 | 기준 정량화(aipc_08) |
| ② 근거 인용 강제 | 인용 출처가 검색된 청크 집합(접지) 밖이면 폐기 | 출력 검증(aipc_01) |
| ③ 사람 검토 격리 | 신뢰도 < **0.60** 이면 수용 대신 '사람 검토 필요' 탭으로 | HITL(aipc_04) |
- \+ **MAX_STEPS=12**(무한루프 차단=툴셋·행동 제한, aipc_06) + **RunLog**(전 스텝 감사추적=전수 로깅, aipc_07) + temperature=0
- 신뢰도 체계: deep 0.85(수용) / outline 0.50(자동 격리) — outline은 정의상 사람 검토로 보냄

## 8. 통제 설계안에 인용되는 기준서 (전량 룰북·코퍼스 경유)
- **감사기준서(KSA)**: 315(문단26·28, 보론3-20 통제활동 유형, 보론6-2 ITGC, A174 사이버·아웃소싱, 확장가능성 A16·A33·A170), 330(통제테스트·8~10·17), 402(서비스조직·SOC 유형1/2), 500(IPE·문단9), 530(표본·보론2), 540(회계추정치), 705(범위제한 4(b)), 1100(벤치마킹 A123·100~102·A128·A129)
- **K-IFRS**: 1109(ECL 5.5.17·B5.5.49~52), 1008(정책 vs 추정 변경), 1115·1113·1002·1116(outline 프로세스)
- **내부회계관리제도 설계·운영 개념체계**: 원칙9~13, 문단30
- 신뢰 등급 노출: verified:true → "원문 대조 완료" 배지(현재 18/18) / false → "원문 대조 전" 배지. **한계를 숨기지 않고 UI에 선언**

## 9. 예상 질문 방어
- "기준서에 AI 조문이 있나?" → 없다. '담당자·사용자'의 유추 적용이며 룰북 근거에 그 한계를 명시했다. 한계를 먼저 인정하는 것이 감사 도메인의 방어 문법.
- "파인튜닝 왜 안 했나" → 기준서는 개정되고, 인용은 검증돼야 함. 파라미터에 녹이면 둘 다 불가. RAG는 룰북 교체만으로 갱신·감사 가능.
- "벡터DB·LangChain 왜 안 썼나" → 규모 대비 과잉. 의존성 하나가 곧 통제 대상 하나. 최소 부품 = 최소 공격면.
- "LLM이 거짓 조항을 만들면?" → 만들어도 게이트②에서 접지 집합 밖 인용은 폐기됨. 애초에 조항은 LLM 출력이 아니라 룰북 필드에서 옴.
- "왜 temperature 0?" → 비결정성이 벤치마킹 붕괴의 원인이라는 게 프로젝트 명제. 내 도구부터 줄여야 일관성 있음.
