# 🧭 AIGC Architect

> **ITGC가 자동통제의 신뢰 기반이었듯, AIGC는 AI 지원 통제의 신뢰 기반이다.**

기업이 AI를 회계 프로세스에 도입할 때, **어떤 재무제표 어서션이 위협받고 어떤 통제가 필요한지**를
산업·조달방식·기업규모에 맞춰 설계해 주는 웹 에이전트입니다.

- **문제의식**: IT통제가 발달하면 통제 의존이 늘고 실증절차는 준다. 자동통제는 "1건 테스트 → 프로그램 로직 일관"이라는 *benchmarking* 논리로 표본을 축소한다. 그런데 **AI 지원 통제는 비결정적(non-deterministic)** 이라 이 전제가 붕괴한다. 그렇다면 AI 통제는 무엇을 근거로 신뢰하는가? → **AIGC(AI General Controls)** 가 그 신뢰 기반이다.
- **AIGC 개념 출처**: 삼일회계법인 제6회 내부통제 미래전략 세미나에서 제시된 개념. 본 프로젝트는 이를 **재무보고 통제 맥락**에서 구체화한다.

---

## 통제의 3층 구조

```
[층 3] AI 지원 통제 — 어서션을 직접 담보
          ▲ 신뢰의 근거
[층 2] AIGC — 모델·프롬프트·학습데이터 통제  ★ 본 프로젝트의 핵심 기여
          ▲ 신뢰의 근거
[층 1] ITGC — AI가 올라탄 시스템 통제 (선결 확인, 게이팅만)
```

**명제: 층 1 없이 층 2 없고, 층 2 없이 층 3 없다.** 도구는 층 1을 게이팅하고, ITGC로 덮이지 않는
고유 영역(비결정성·프롬프트·학습데이터)인 층 2에 집중한다.

## ★ 자기참조 설계 (이 프로젝트의 최대 무기)

> **이 도구의 Harness는, 이 도구가 설계하는 AIGC 통제를 스스로에게 적용한 것이다.**

| 이 도구가 설계하는 AIGC 통제 | 이 도구의 Harness 구현 | 코드 위치 |
|---|---|---|
| 설명가능성·근거 추적 | 근거 조항 인용 강제 — 검색된 지식에 없는 인용은 출력에서 차단 | `harness.enforce` 게이트 2 |
| Human-in-the-loop | 신뢰도 낮은 판정은 확정 않고 ⚠️ 격리 | `harness.enforce` 게이트 3 |
| 감사추적 | 실행 로그 — 스킬 호출 순서·검색 청크·소요시간 전량 기록 | `harness.RunLog` |
| 할루시네이션 통제 | 출력 스키마 검증(pydantic), 미검증 근거는 "원문 대조 전" 배지 | `schemas.ControlCard`, 게이트 1 |
| 비결정성 관리 | temperature 고정 | `llm.TEMPERATURE = 0.0` |
| 무한루프 방지 | 스텝 상한 | `harness.MAX_STEPS` |

즉, 감사인이 AI 통제에 요구하는 바로 그 통제들을, 이 AI 도구가 자기 자신에게 부과한다.

## 아키텍처

```
aigc-architect/
├── rulebook/                    # 도메인 지식 (도구의 분기 로직)
│   ├── industry_profiles.yaml   # 산업 5 × 프로세스 13, 유의성 등급 + 판단 근거
│   └── control_ledger.yaml      # ITGC 게이트 4 + 통제 (deep 2, outline 8)
├── corpus/                      # RAG 원문 (감사기준서 발췌 등 — 사용자가 추가)
├── src/
│   ├── schemas.py    # Harness #1 — pydantic 출력 스키마 (근거 필수)
│   ├── llm.py        # Claude 생성 + 임베딩 (temperature 고정, 우아한 폴백)
│   ├── rulebook.py   # 룰북 로딩 + RAG 청킹
│   ├── rag.py        # numpy 코사인 검색 (벡터DB 미사용)
│   ├── skills.py     # 능력 단위 6개 함수
│   ├── agent.py      # 조건 분기 오케스트레이션 (스텝별 진행)
│   └── app.py        # Streamlit UI
└── tests/smoke_test.py          # 헤드리스 파이프라인 검증
```

### Skills / Agent / Harness

- **Skills** — 에이전트가 쥐는 능력 단위 모듈(함수). `check_itgc_gate`, `identify_significant_accounts`, `map_assertion_risk`, `retrieve_controls`, `contextualize_controls`, `generate_auditor_view`.
- **Agent** — 스킬을 *조건에 따라* 순서 판단하며 호출. 살아있는 분기: ITGC 실패 → 하위 중단 / `ai_performed_control` 선택 → 비결정성 통제(`aipc_01`,`aipc_02`) 강제 포함 + 추가 RAG / `critical` → 상술, `routine` → 요약 / **산업별로 프로세스 목록이 실제로 달라진다.**
- **Harness** — 에이전트를 실행·감시·제어하는 틀. 위 자기참조 표 참조.

## RAG 설계 근거

### 왜 벡터DB를 쓰지 않았는가

코퍼스가 작다(청크 수백 개 규모). 임베딩을 **L2 정규화하면 코사인 유사도 = 내적**이므로,
`(질문벡터) · (청크행렬.T)` **행렬곱 한 번**으로 전수 유사도 계산이 끝난다.
HNSW 등 근사최근접(ANN) 인덱스의 이득은 수십만 벡터 이상에서 나타나며, 그 이하에서는
인덱스 구축·유지 비용이 오히려 손해다. → **numpy 코사인으로 충분하다.**

### "코퍼스가 작으면 전체를 그냥 프롬프트에 넣어도 되지 않나?"

맞다. 그리고 그 판단이 설계 역량의 핵심이다.

- **전체 주입(stuff)이 나은 조건**: 문서 총량이 컨텍스트 윈도우에 안전하게 들어가고, 질문이
  문서 전반을 교차 참조해야 하며, 토큰 비용을 감당할 수 있을 때.
- **RAG가 나은 조건**: 문서가 커져 컨텍스트를 초과하거나, 근거를 *어느 청크에서 왔는지*
  추적·인용해야 할 때(감사추적!), 무관한 맥락을 배제해 정확도·비용을 최적화할 때.

이 도구는 후자를 택했다. **근거 추적성**이 프로젝트의 명제(설명가능성·감사추적)와 직결되기 때문이다.
검색된 청크의 출처가 곧 Harness의 인용 접지(grounding) 기준이 된다.

## 보안 · 성능 트레이드오프 (온프레미스 / SLM 검토)

재무데이터는 민감하다. 실무 배포 시 고려해야 할 축:

- **외부 API(본 데모 기준)**: 최신 대형 모델의 추론 품질. 단, 데이터가 외부로 나가므로
  전송·보관 통제, 공급사 SOC 보고서, 데이터 잔류 정책 검토 필요.
- **온프레미스 / 프라이빗 배포**: 데이터가 경계를 넘지 않음. 인프라·운영 비용 상승.
- **경량 온프레미스 LLM(SLM)**: 분류·추출 등 구조화 태스크는 소형 모델로도 충분한 경우가 많다.
  민감 데이터를 내부에 두면서 비용을 낮추는 절충. 대신 복잡 추론·긴 문맥에서 품질 저하 가능.

본 도구는 `llm.py`에서 제공자를 추상화했다. 조직의 데이터 등급·규제에 따라 백엔드를
교체할 수 있도록 설계했으며, 이는 통제 설계의 일부다(모델 조달 방식이 통제 설계에 반영됨).

## 실행 방법

```bash
# 1) 의존성
pip install -r requirements.txt

# 2) (선택) API 키 — 없으면 룰북 템플릿 + 로컬 TF-IDF로 자동 대체되어 그대로 동작
cp .env.example .env    # ANTHROPIC_API_KEY, VOYAGE_API_KEY 입력

# 3) 실행
streamlit run src/app.py
#   또는
./run.sh

# 4) 파이프라인 검증 (헤드리스)
python tests/smoke_test.py
```

**우아한 성능저하**: `ANTHROPIC_API_KEY`가 없으면 통제 구체화·감사인 관점은 룰북 기반
템플릿으로, `VOYAGE_API_KEY`가 없으면 임베딩은 로컬 TF-IDF 코사인으로 대체된다.
덕분에 API 키 없이도 Agent·Harness·RAG의 동작을 end-to-end로 시연할 수 있다(데모 GIF 용이).

## 한계 선언

- **기준서 조항 미검증 13건**: `control_ledger.yaml`의 `evidence.verified: false` 항목은
  조항 번호가 아직 원문 대조 전이다. 도구는 이를 **"원문 대조 전" 배지**로 정직하게 표시하며,
  대조·상술 전 통제는 신뢰도를 낮춰 **사람 검토 큐로 격리**한다.
- **도구는 조항을 지어내지 않는다.** 근거는 룰북에서만 오며, 검색된 지식에 없는 인용은 차단된다.
- **실명 기업을 판정하지 않는다.** 도구는 "이 프로필이면 이런 통제가 필요하다"를 말하지,
  "A사는 미비하다"를 말하지 않는다.
- **깊이 우선**: `depth: deep` 2개(`ai_performed_control`=프로젝트 명제, `ecl_estimation`=
  감사기준서 540 영역)가 방어의 핵심이다. `outline` 8개는 골격이며 확장 예정이다.
- **다루지 않는 것**: 대고객 챗봇·마케팅 추천 등 재무보고와 무관한 AI. 개인정보·소비자보호 축은 별도.
