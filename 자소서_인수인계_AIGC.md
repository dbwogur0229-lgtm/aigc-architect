## 1. 프로젝트 한 줄과 핵심 명제
기업이 AI를 회계 프로세스에 도입할 때 어떤 경영진주장이 위협받고 어떤 통제가 필요한지 설계해주는 LLM 기반 RAG 에이전트(Streamlit 배포 완료).

핵심 명제: "ITGC가 자동통제 신뢰의 기반이었듯, **AIGC(AI General Controls)**가 AI 지원 통제 신뢰의 새 기반이다. AI는 비결정적이라 자동통제의 벤치마킹(Sample=1) 논리가 성립하지 않기 때문이다(KSA 1100 A123·100~102)." 3층 구조: ITGC(게이트) → AIGC(핵심) → 프로세스 통제.

## 2. 자소서에 쓸 수 있는 검증된 수치 (이 밖의 수치 창작 금지)
- RAG 130청크(룰북 46+코퍼스 84), 벡터DB 미사용(numpy 코사인), 임베딩 Voyage→TF-IDF 폴백
- 근거 조항 **18건 전량 기준서 원문(974p) 대조 검증**, 그 과정에서 **초기 가설 4건 교정**
- ITGC 게이트 7체크·4카테고리 / 5산업×13프로세스 / deep 2(통제 14개)·outline 8
- 에이전트 5스텝, 스킬 6개(결정론적 Python 함수), Harness 3중 게이트, 스텝 상한 12
- 신뢰도 deep 0.85/outline 0.50/사람검토 임계 0.60 — 룰북 depth에서 결정론적 산출
- temperature=0 고정, API 키 없는 폴백 모드로 배포 = "LLM 없이도 결론 성립"의 증거

## 3. 확정 문언 (틀리면 감점, 검증 과정에서 확정된 표현)
- ❌ "LLM을 만들었다/개발했다" → ⭕ "LLM 기반 RAG 에이전트를 만들었다", "LLM은 부품으로 조달, 지식기반·검색·오케스트레이션·실행통제를 직접 설계"
- ❌ 전수 검증이 표본을 "대체" → ⭕ "**보완**" (KSA 500 A64 단서 때문. 상시 모니터링도 독립적 평가의 조정·보완)
- ❌ "비결정성 제거" → ⭕ "최소화"
- ❌ "기준서가 AI 통제를 규정" → ⭕ 기준서에 AI 명시 조문 없음, "유추 적용"임을 인정
- 이 도구는 판정이 아니라 **설계 권고**. 실명 기업 판정 언급 금지.

## 4. 스토리 아크 후보 (문항 유형별)
- **A. 가설이 틀렸던 이야기 (실패·수정 문항 최적)**: 신규 통제 5건의 근거를 원문 대조했더니 4건에서 초기 가설이 어긋남 — 접근보안 근거를 KSA 1100으로 추정했으나 원문 확인 결과 1100 문단 35가 315 보론 6으로 위임. 반대 문구(500 A64 단서, 개념체계 A68)까지 룰북에 병기해 과장을 막음. 메시지: "검증은 내 주장이 틀렸음을 발견할 때 가치가 있다" — 감사 마인드 그 자체.
- **B. 자기참조 설계 (직무 이해 문항)**: 도구가 기업에 권고하는 AIGC 통제를 도구의 Harness가 스스로에게 적용(인용 강제·사람검토 격리·실행로그·스텝상한·temp 0). 요건 ②의 직접 증명.
- **C. LLM 없이도 돌아가는 배포 (기술 판단 문항)**: 감사 도메인은 검증가능성이 생명 → 지식을 LLM 파라미터가 아닌 사람이 검증한 룰북에 두고, LLM은 재기술만. 할루시네이션이 결론을 오염시킬 경로를 구조적으로 차단. 배포본(폴백 모드)이 살아있는 증거.
- **D. 깊이>넓이 (우선순위·판단 문항)**: 13개 프로세스 전부 얕게 대신 deep 2개(AI 수행 통제 + ECL)를 방어 가능한 깊이로. 덜 검증된 outline은 자동으로 '사람 검토' 격리 — 한계를 UI에 선언하는 설계.
- **E. 외부 검증 (소통·검증 문항)**: 삼일 PwC 세미나(공개)와 대조해 전수 패러다임·과정 전체 관리·Trust by Design 합치 확인. 7/18 현직 회계사 미팅으로 명제 검증. → 미팅 후 결과 업데이트 필요.
